# Phase 0A: Human-Readable Business Numbering Plan

## 1. Purpose

Define the future human-readable business numbering system for CBM Trade OS before implementing database fields, API logic, UI flows, or numbering utilities.

This document is planning only. It does not implement numbering code or modify the database.

## 2. Why CBM Trade OS Needs Human-Readable Business Codes

CBM Trade OS will manage customers, suppliers, inquiries, projects, RFQs, quotations, PI, orders, documents, attachments, communication records, tasks, knowledge articles, AI tasks, and approval reviews.

Human-readable codes make records easier to search, discuss, verify, archive, export, and connect across daily foreign trade work.

## 3. Core Principles

- Codes are stable after creation.
- Codes are unique and searchable.
- Codes show initial business context.
- Codes should not be changed when business fields change later.
- Detailed business attributes should still live in fields and tags, not only in the code.
- AI may generate code suggestions, but system rules control final code generation.
- UUID primary keys should still exist separately from human-readable business codes.

## 4. Customer Company Code Format

Format:

```text
CUST-{COUNTRY}-{TYPE}-{SOURCE}-{YEAR}-{SEQ}
```

Example:

```text
CUST-PE-DIST-ALI-2026-0001
```

Meaning:

- `CUST`: customer company
- `PE`: country code, Peru
- `DIST`: customer type, distributor
- `ALI`: source, Alibaba
- `2026`: created year
- `0001`: sequence number

## 5. Customer Contact Code Format

Format:

```text
{CUSTOMER_CODE}-C{SEQ}
```

Example:

```text
CUST-PE-DIST-ALI-2026-0001-C01
```

A customer company may have many contacts. Contact numbering should be stable even if contact names or roles change later.

## 6. Supplier Company Code Format

Format:

```text
SUP-{COUNTRY}-{CATEGORY}-{SOURCE}-{YEAR}-{SEQ}
```

Example:

```text
SUP-CN-ALU-MAN-2026-0001
```

Meaning:

- `SUP`: supplier company
- `CN`: country code, China
- `ALU`: supplier category, aluminum
- `MAN`: source, manual entry
- `2026`: created year
- `0001`: sequence number

## 7. Supplier Contact Code Format

Format:

```text
{SUPPLIER_CODE}-C{SEQ}
```

Example:

```text
SUP-CN-ALU-MAN-2026-0001-C01
```

```text
INQ-{COUNTRY}-{YEAR}-{SEQ}
PRJ-{COUNTRY}-{YEAR}-{SEQ}
RFQ-{COUNTRY}-{YEAR}-{SEQ}
SQUO-{COUNTRY}-{YEAR}-{SEQ}

DOC-{YEAR}-{SEQ}
ATT-{YEAR}-{SEQ}

TASK-{YEAR}-{SEQ}

```

Notes:

- `SQUO` means supplier quote.
- `QUO` means customer quotation.
- `COM` means communication thread or message reference, depending on future design.
- `APR` means approval review.

## 9. Suggested Country Code Rule

Use ISO two-letter country codes when possible.

Examples:

- `CN`: China
- `US`: United States
- `PA`: Panama
- `PE`: Peru
- `AE`: United Arab Emirates
- `ES`: Spain
- `MX`: Mexico
- `BR`: Brazil

If country is unknown, use `XX` until confirmed. Do not rename existing codes after country is later clarified.

## 13. Duplicate Detection And Code Generation Workflow

Recommended workflow:

1. User or AI proposes a new business object.
2. System searches for possible duplicates using name, email, WhatsApp, website, country, alias, and existing codes.
3. If possible duplicate exists, show candidate records for human review.
4. If no duplicate is found, system generates the next code using controlled rules.
5. User confirms creation in a manual write flow.
6. Code becomes stable and should not be changed automatically.

AI may suggest type, source, country, and category, but the final code should be generated by deterministic system rules.

## 14. How AI Agents Should Use Numbering

AI agents should:

- display existing codes when summarizing records
- use codes to avoid mixing customers, suppliers, inquiries, or documents
- suggest likely country/type/source/category before creation
- ask for human review when classification is uncertain
- never overwrite an existing code automatically
- never create official commercial documents only from a code without checking linked data

## 15. What Should NOT Be Implemented Yet In Phase 0A

Do not implement yet:

- database columns for codes
- migrations
- code generation utilities
- API endpoints for numbering
- UI create flows
- automatic duplicate merge
- automatic customer or supplier creation
- OpenAI/model-based code generation
- official quotation, PI, order, shipment, payment, or document automation

## 16. Future Implementation Roadmap

Recommended order:

1. Finalize naming rules and code dictionaries.
2. Add tests for pure code formatting rules.
3. Add a deterministic numbering utility.
4. Add code preview only in admin UI.
5. Add database fields through an approved migration.
6. Add API-only read/write support for low-risk objects.
7. Add duplicate detection preview.
8. Add manual create flows with human confirmation.
9. Allow AI to suggest code components while system rules generate final codes.

## 17. Acceptance Criteria For Future Implementation

Future implementation should be accepted only if:

- codes are unique and stable after creation
- code generation is deterministic and testable
- UUID primary keys remain separate
- AI can suggest code components but cannot bypass system rules
- duplicate candidates are shown before creating new companies or suppliers
- unknown fields use safe placeholders such as `XX`, `UNK`, or `OTH`
- no existing code is changed automatically when business fields change later
- no business commitment is created by numbering alone
