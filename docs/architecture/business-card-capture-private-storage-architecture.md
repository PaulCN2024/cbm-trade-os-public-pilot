# Business Card Capture Private Storage Architecture

## Purpose

This plan defines how future business card image files should be stored securely before real upload is enabled.

It is planning only. It does not create a bucket, modify Supabase Storage, change schema, add SQL, implement upload, enable OCR, parse images, create customers, send messages, or execute business actions.

## Why Private Storage Is Required

Business card images may contain:

- name
- company
- title
- phone
- email
- WhatsApp
- address
- website
- QR code
- private contact info

Because these files can include personal contact details and business relationship context, they must not be publicly accessible.

## Recommended Bucket Model

Future private Supabase Storage bucket proposal:

```text
business-card-captures
```

Rules:

- private bucket
- authenticated access only
- no public URL
- no public CDN
- no direct anonymous access
- signed preview URL only if needed later
- no service-role key in frontend code

This bucket should hold original business-card/contact images only. OCR output, customer-profile drafts, duplicate checks, and follow-up drafts should remain in reviewed database records, not in public file metadata.

## File Path Strategy

Proposed future object path format:

```text
business-card-captures/{year}/{month}/{capture_source_id}/{safe_filename}
```

Rules:

- sanitize filename before storage
- no raw customer email in path
- no phone number in path
- no WhatsApp number in path
- no personal name required in path
- no private note text in path
- UUID-based folder names are preferred
- path should be generated by a protected backend flow, not trusted from browser input

Example safe path shape:

```text
business-card-captures/2026/06/4b7f0a2e-0d4c-4d3f-9c8c-5c15f0c8b7e2/card-image.webp
```

## Metadata Linkage

The storage object should link to `card_capture_sources.original_file_id` or a future internal storage reference such as `storage_object_path`.

Future `card_capture_sources` may need fields such as:

- `storage_bucket`
- `storage_path`
- `file_mime_type`
- `file_size_bytes`
- `storage_status`
- `preview_status`

Do not create a migration from this document. These fields require a separate schema implementation task and approval.

## Upload Lifecycle

Planned future lifecycle:

```text
file selected
-> client-side validation
-> protected upload endpoint later
-> private bucket object
-> card_capture_sources record
-> extraction pending
-> OCR later
-> review
-> approve/reject/archive/delete
```

Important boundaries:

- Upload only creates a source for review.
- Upload must not create a customer.
- Upload must not send a message.
- Upload must not trigger OCR unless a later OCR workflow is explicitly approved.
- Upload must not create quotation, PI, order, payment, production, or shipment records.

## Access Layers

The future access model should have these layers:

1. Browser UI
   - validates file type and size for operator feedback
   - does not receive privileged storage credentials
   - does not build final storage path alone

2. Protected upload endpoint
   - requires authenticated admin session
   - validates MIME type, extension, size, and basic image sanity
   - generates safe storage path
   - creates or links `card_capture_sources`

3. Private storage bucket
   - no public read
   - no anonymous access
   - object access mediated by policy or protected backend route

4. Preview access
   - short-lived signed URL or server-mediated preview only after review
   - no permanent public image links in Admin UI state

5. Future role layer
   - owner/admin/reviewer/readonly roles before team use
   - role planning must happen before broader access

## Non-goals

- No upload now.
- No signed URL implementation now.
- No OCR now.
- No image parsing now.
- No schema migration now.
- No API implementation now.
