# Business Card Capture Upload Safety Architecture

## Purpose

This document defines how future Business Card Capture image upload should be implemented safely before enabling real uploads.

The goal is to prepare an upload architecture that can later support business-card images, exhibition cards, WhatsApp contact screenshots, and referral contact images without exposing private contact data, creating customers automatically, or triggering outbound messages.

This is planning only. It does not implement upload, OCR, image parsing, storage buckets, API routes, schema changes, SQL, customer creation, message sending, or business execution.

## Why Upload Is Sensitive

Business card images may contain personal and business contact information:

- name
- job title
- company identity
- phone
- Email
- WhatsApp
- website
- office address
- QR codes
- private notes written on a card

Even when the card is business-related, the image and extracted fields should be treated as sensitive internal data. The system must avoid public URLs, accidental sharing, automatic customer creation, and unreviewed outreach.

## Future Upload Workflow

Planned future flow:

```text
user selects image
-> client validates type/size
-> file stored in protected storage
-> capture source record created
-> OCR/vision extraction later
-> extraction result draft
-> customer profile draft
-> duplicate check
-> human review
-> approved customer creation later
```

Workflow principles:

- Upload only creates a source for review; it must not create a customer.
- OCR/vision output must become an extraction result draft, not trusted customer data.
- Duplicate checks must run before any approved customer creation workflow.
- Customer creation and follow-up sending must remain separate approval-gated actions.

## File Type Restrictions

Allowed first-stage file types:

- `image/jpeg`
- `image/png`
- `image/webp`

Possible later file type, only after explicit approval:

- `application/pdf`

Disallowed first-stage file types:

- executable files
- archives such as `.zip`, `.rar`, `.7z`
- scripts such as `.js`, `.sh`, `.py`, `.bat`
- unknown binary formats
- large videos
- Office documents in the first upload stage

Validation should check both MIME type and extension. A file must pass both checks before upload.

## Storage Plan

Future storage should:

- use a private bucket
- avoid public file URLs
- require authenticated access
- store minimal metadata
- link the stored object to `card_capture_sources`
- avoid exposing service-role credentials in client code
- avoid showing raw storage paths in the Admin UI unless explicitly needed for internal debugging
- support later retention/deletion decisions

The first real upload implementation should not expose signed URLs in persistent UI state. If previews need signed URLs, they should be short-lived and generated by an authenticated backend route after policy review.

## Access-control Plan

Access rules:

- authenticated internal users only
- no anonymous read
- no public read
- no customer-facing direct access
- future role-based access for operator/admin separation
- extracted contact data remains review-only until approved

Sensitive contact extraction requires review before it can affect:

- Customer Center official records
- follow-up drafts
- outbound Email / WhatsApp messages
- prospecting or marketing lists
- quotation, PI, order, payment, production, or shipment workflows

## Error Handling

Future UI/API should handle:

- invalid file type
- invalid extension
- file too large
- upload failed
- storage unavailable
- unauthorized session
- OCR failed
- low confidence extraction
- duplicate detected
- manual review required

Error states should be business-safe:

- Do not expose storage paths, signed URLs, secrets, or provider payloads.
- Do not imply customer creation happened.
- Do not retry endlessly.
- Keep failed uploads reviewable or discardable according to retention policy.

## Non-goals

This plan does not authorize:

- upload implementation now
- OCR now
- image parsing now
- storage bucket creation now
- schema changes now
- SQL execution now
- customer creation now
- follow-up sending now
- public file access
- automatic customer verification
- business execution

## Future Approval Gate

Before enabling real upload, the project should approve:

- private bucket and access policy
- accepted MIME/extension list
- maximum file size
- signed URL behavior
- metadata fields
- retention/deletion behavior
- audit expectations
- failure and retry handling
- user-facing Chinese safety wording
